# Design Decisions Summary

This document logs the consensus-based decisions made during Kite design sessions. Detailed reasoning for major architectural shifts can be found in the `docs/adrs/` directory.

| Date | Decision | Context / Motivation | Associated ADR / Task |
|------|----------|----------------------|-----------------------|
| 2026-05-19 | Established Design Workflow | Need a structured way to run consensus-based design sessions without writing code directly. | N/A |
| 2026-05-19 | Virtual Markers for List Layout | Needed a way to render list bullets/numbers without violating the unified render box rule or causing clipping in the terminal grid. | ADR 001 |
| 2026-05-19 | Logical DOM Components for Lists | Introduced `ul`, `ol`, and `li` components. Decided to use `PaddingLeft` for terminal indentation and explicitly map `ListStyleType` as an inheritable style property. | Task: TSK-002 |
| 2026-05-19 | Table Layout and Fault Tolerance | Designed a two-pass algorithm for tables supporting ColSpan/RowSpan. Resolved to handle malformed tables strictly within the layout algorithm (like IFC anonymous blocks) to keep the engine sync phase pure. | ADR 002 |
| 2026-05-19 | Declarative UI API and DOM Adoption | Moved to a declarative API (`Box(Text("Hello"))`). Required modifying `dom.AppendChild` to implicitly adopt detached nodes from different documents. | ADR 003 |
| 2026-05-19 | Logical DOM Owns Element Styles | Moved ownership of `RawStyle` and `DefaultStyle` exclusively to the DOM elements. Render objects now proxy these calls. Maintains CSS cascade layering while avoiding state duplication. | Task: TSK-007 |
| 2026-05-19 | Table Grouping Elements | Decided to support `thead`, `tbody`, and `tfoot` logic in the table layout algorithm. This ensures perfect column alignment between sticky headers and data rows, and matches standard CSS behavior. Required adding anonymous row groups for fault tolerance. | ADR 002 |
| 2026-05-19 | Border Fluent API & Rendering | The `paint` engine's border rendering will be rewritten to respect per-edge widths, styles, and colors using defined Unicode glyph maps. To improve DevX, `style.Border` will act as an immutable fluent builder itself (e.g., `style.SingleBorder().Color(red)`), and we will create a type alias or property rename to avoid collision with `Optional[Border]`. | Pluralized struct fields (Edges, Styles, Colors) to allow singular fluent modifiers. Width changed to boolean Edges to reflect terminal constraints. | Task: TSK-009 |
| 2026-05-19 | Table Border Collapsing & Grid Painter | In a TUI, table cell borders must physically overlap by 1 cell and be drawn collectively to form correct Unicode intersections (e.g., `┼`, `├`). We will implement a specialized `paintTableGrid` pass triggered when `DisplayTable` is encountered in the paint engine, suppressing standard border painting for cells. | Task: TSK-010 |
| 2026-05-19 | Global Border Intersection Resolver | We replaced the `Table Grid Painter` with a global screen-space post-processing pass. The paint engine will flag all border cells with `FlagIsBorder`. At the end of the frame, a single $O(W \times H)$ pass resolves all intersecting borders (tables, sidebars, cards) into perfect Unicode junctions by checking neighbors. | Task: TSK-010 |
| 2026-05-19 | Table Layout Implicit Border Overlap | To support the Global Border Intersection Resolver, the Table Layout algorithms (`TableAlgorithm` and `TableRowAlgorithm`) must physically overlap adjacent cells by 1 coordinate space if they share a border. This ensures the paint engine plots their border glyphs on the exact same framebuffer cell, allowing the post-processor to correctly form cross-junctions (`┼`). | Merged into TSK-012 | Task: TSK-012 |
| 2026-05-19 | Table Layout Builder Pattern | To manage the complexity of two-pass table layout (grid sizing, spanning, grouping, and implicit border collapse), we will introduce a dedicated `TableFragmentBuilder` (or `GridSizingBuilder`) in the layout package. This builder will encapsulate the matrix math and state mutations, keeping `TableAlgorithm` focused purely on traversal logic. | Task: TSK-012 |
| 2026-05-19 | Layout Builder Standardization (Flex) | To enforce clean architecture across the Layout Engine, complex layout algorithms must separate coordination from state mutation via the Builder Pattern. Following Tables, the `FlexAlgorithm` will be refactored to use a `FlexLineBuilder` to handle item chunking, fraction distribution, and alignment math. | ADR 005 / TSK-013 |
| 2026-05-19 | Global Border Style Metadata | Replaced the boolean `FlagIsBorder` mask with an explicit `BorderStyle` enum on `paint.Cell`. This allows the border intersection resolver to handle mixed-style intersections using a "Heaviest Style Wins" precedence rule, removing the need for string sniffing. | ADR 006 / TSK-014 |
| 2026-05-20 | Input & TextArea Architecture | Input fields require scrollable text and cursor positioning without polluting the DOM with physical layout math. | ADR 007 / Tasks TSK-015 to TSK-018 |
| 2026-05-21 | UA Shadow Subtree for Replaced Elements | The "Replaced Element via Direct Casting" pattern (ADR-007) does not scale to compound widgets (checkbox, radio, select, slider) and duplicates styling guards. Replaced and compound elements now own a closed UA shadow subtree (engine-walked, dispatch-invisible, never focusable) via an internal `uaRoot`. Author code cannot reach it; `outer` retargets dispatch to the host. Supersedes ADR-007. | ADR 009 / TSK-018, TSK-024, TSK-025 |
| 2026-05-21 | Intrinsic Style Layer | UA-mandated properties (e.g., `display: inline-block`, `overflow: clip`, `white-space: pre-wrap`) need to be enforced declaratively above author styles in the cascade. Adds `IntrinsicStyle()` to `dom.Element` as the strongest cascade layer (origin `OriginUserAgent`); `DefaultStyle()` remains the weakest, author-overridable layer. | ADR 010 / TSK-022 |
| 2026-05-21 | Paint-Phase Overflow Clipping | The `style.Overflow` enum was unread by the paint engine; `Surface.Clip()` existed but had zero callers. Paint now applies content-box clipping during recursion for any fragment whose `OverflowX/Y` is `Hidden`, `Clip`, `Scroll`, or `Auto`. The fragment's own background and border paint unclipped at the parent's level; clipping applies to descendants only. Reuses existing `clippedSurface` machinery. | ADR 011 / TSK-027 |
| 2026-05-21 | Generic Scroll Offset on DOM Elements | Replaces the bespoke `scrollX`/`scrollY` machinery on `<input>` and `<textarea>` with a uniform browser-style scroll model: every `dom.Element` exposes `Scroll`, `ScrollTo`, `ScrollBy`; lazy `*scrollState` allocated only for scroll containers; paint clamps on read and translates child origins; default `Scrollable` wheel handler vended by the framework; new `event.EventScroll`. Adds `OverflowAuto` to the enum and a fluent `.Overflow(v)` shorthand on `style.Style` (keeps the X/Y split for asymmetric cases). | ADR 012 / TSK-028 |
| 2026-05-21 | Unified Text Control Base | Centralized duplicated caret positioning, scrolling math, and core event handling from `InputElement` and `TextAreaElement` into a shared `textControlBase` struct. Relies purely on the standard layout fragment bounds and `dom.Element.Scroll()` to handle viewports without creating specialized layout nodes, enabling future scrollbar UI support automatically. | ADR 013 / TSK-029 |
| 2026-05-21 | Kite DevTools Architecture | Introduced `kite/devtools` to prevent core bloat. Features include an SSE-based Web DOM Inspector, an in-terminal "X-Ray Mode" for visual layout debugging, and a headless `testenv` utilizing the existing `backend/mock` for DOM queries, input simulation, golden testing, and HTML visual dumps. | ADR 014 / Tasks TSK-030 to TSK-032 |
| 2026-05-22 | Event Coalescing and Deferred Scroll | Decoupled the backend input frequency from the 60FPS render frame. Incoming input events are buffered and coalesced (aggregating Wheel deltas, squashing MouseMoves) just before the frame starts. Heavy layout recalculations triggered by scrolling (like text cursor adjustments) are deferred to the layout/paint lifecycle rather than running synchronously in event handlers. | ADR 015 |
| 2026-05-22 | Customizable Visual Scrollbars | Replaced the vague `ScrollBox` roadmap concept with an explicit, CSS-like `style.Scrollbar` struct. Layout dynamically reserves space if `Scrollbar.X/Y` is true and `Overflow` dictates scrolling. Paint independently draws customizable tracks and thumbs within the reserved space based on the element's dynamic scroll offset. | Task: TSK-036 |
| 2026-05-22 | Interactive Form Controls | Formalized architecture for Button, Checkbox, RadioGroup, and Select. Checkbox/Radio use UA Shadow Subtrees for glyphs. RadioGroup enforces a strict parent-child structural relationship to avoid DOM-wide lookups. Select utilizes the existing `Overlay` system to spawn anchored dropdowns and pushes local Focus Scopes. | ADR 016 |
| 2026-05-22 | Strict TUI Box Model | Departed from CSS defaults by permanently locking the framework to a strict `border-box` sizing model with absolutely no margin collapsing. This guarantees predictable, additive math for discrete terminal grids. | ADR 017 |
| 2026-05-22 | Layout ContainingSpace & ContainerSpace | Introduced two explicit spatial concepts into `ConstraintSpace`: `ContainingSpace` (parent's border-box, used for percentage resolution) and `ContainerSpace` (parent's inner/content-box, the available space for children). Replaces `PercentageResolutionSize` and eliminates duplicated decoration math across all layout algorithms. A shared `BuildChildSpace` function centralizes child constraint generation. | ADR 018 / TSK-041 |
| 2026-05-23 | Profiler Architecture (Hybrid) | Built-in DevTools profiling. Uses a Pipeline Decorator for engine phases and an inlineable `TraceContext` struct for deep granular timings (e.g., specific layout algorithms) with zero allocations when disabled. Configurable via `engine.WithProfiler(true)` for startup waterfalls. Outputs Chrome Trace Event format (JSON). | ADR 019 |
| 2026-05-23 | DevTools Frontend Architecture | Migrated the DevTools Inspector frontend from a plain HTML/Vanilla JS file to a Preact + Vite bundled application. This allows building complex data visualizations (like Flamecharts) using component-based architecture while Vite is configured to output a single HTML file for zero-dependency Go embedding. | ADR 020 / TSK-044 |
| 2026-05-23 | Animation Architecture | Implemented an imperative, package-based animation system (`animation.Tween[T]`) to keep the style engine stateless. Animations are registered with the `Engine`, which ticks them during the frame loop and self-schedules via `RequestFrame()` until they complete. | ADR 021 |
| 2026-05-24 | Text Selection & Clipboard | Designed a logical Selection API mirroring browser specs. Adopted a "Push Model / Independent List" architecture for rendering, where logical text ranges are pre-resolved into physical rects and fed to the paint engine as an overlay mask, maintaining 60FPS layout cache performance. Integrated system clipboard via existing event synthesizer bridge. | ADR 022 |
| 2026-05-24 | Input/Textarea Selection & Clipboard | Replaced elements with closed UA subtrees (`<input>`, `<textarea>`) manage their own logical selection state (`SelectionStart`, `SelectionEnd`). When active, they bridge this state to the global `dom.Selection` to reuse the 60FPS paint masking. They intercept standard `Copy`, `Cut`, and `Paste` events natively to manipulate their internal values and OS clipboard without exposing their shadow DOM text nodes. | ADR 022 extension |
| 2026-05-24 | CSS Grid Layout Algorithm | Designed a 3-phase CSS Grid layout algorithm (Measure Tracks -> Auto-Placement -> Layout) constrained by `layout.Builder` patterns to maintain caching efficiency. Scoped initial implementation to explicit tracks, `fr` units, and auto-placement (deferring `minmax()`). Grid animations are supported generically via an explicit `animation.Interpolator[[]GridTrackSize]` plugged into the existing `Tween[T]` system. | ADR 023 |
| 2026-05-24 | Terminal Extension Architecture | Introduced `backend.TerminalExtension` to allow interception of raw terminal sequences (CSI/OSC/APC) before standard synthesis. Enables support for complex protocols like Kitty secure clipboard transfer (OSC 5522) and advanced graphics without bloating the core backend. Engine manages a registry and provides direct TTY access to extensions during startup. | Task: TSK-049 |
| 2026-05-24 | Advanced Clipboard Integration | Enhanced `ClipboardEvent` with multi-MIME `Items` map (DataTransfer-like). Implemented global document-level copy/paste handlers with OSC 52 system synchronization. Text components (`Input`, `TextArea`) now intercept `EventPaste` for cursor-relative insertion and allow bubbling for global feedback. | Task: TSK-049 |
| 2026-05-25 | `kitex` Reactive UI Framework | Introduced `extras/kitex`, a React-like VDOM framework built on top of the Kite DOM. Features a functional component API (`react.FC<P>`), a VDOM reconciler, implicit hook context via an internal execution stack, and a dedicated declarative API (`element` package) for modifier-based DOM nodes. | ADR 024 |
| 2026-05-25 | `kitex` Ref Standardization | Standardized component/DOM references in `kitex`. Renamed the internal `liveNode` to `ref`, and the hook tracking pointer from `ref` to `componentRef` to clarify terminology. Introduced the user-facing `Ref[T]` type (a pointer to a struct holding `Current T`) along with `UseRef` and `CreateRef` hooks, using an unexported `refSetter` interface to safely attach real `dom.Node` instances to elements during the `Instantiate` and `Update` reconciliation phases. | Task: TSK-059 |
| 2026-05-25 | `kitex` DevTools Integration | Designed an out-of-band DevTools integration for `kitex` to prevent polluting the core `dom` package. Added an `Extension` interface to `devtools/inspector`. Added an opt-in `kitex.EnableDevMode` flag and zero-overhead `runtime.Caller` declaration source tracking to `ComponentNode`. Added a standalone `kitex.BuildDevToolsSnapshot()` function to export private VDOM states securely. Created `kitexdt` package to bridge the snapshot into the DevTools Extension API. Designed a Preact frontend "Components" tab with highlighting cross-linking between VDOM, DOM, and Layout Fragments. | Task: TSK-060 |
| 2026-05-25 | `kitex` Automatic Component Memoization | Implemented JIT component memoization similar to React Compiler. To prevent unnecessary VDOM allocations and reconciliations, `ComponentNode` re-renders are automatically skipped using a depth-limited reflection check (`deepEqualProps`) on properties. To avoid reflection overhead on simple components, memoization is only activated if a component's VDOM tree exceeds a complexity threshold (e.g., > 5 nodes). The complexity score is calculated in $O(1)$ time per component using a bottom-up aggregation via an internal `complexity() int` interface during tree construction. A `UseMemo` hook was also introduced. | Task: TSK-061 |
| 2026-05-26 | `kitex` Effect Hooks (`UseEffect`, `UseLayoutEffect`) | Introduced side-effect hooks with a dual-variant API: `UseEffect(fn func(), deps)` / `UseEffectCleanup(fn func() func(), deps)` for post-commit effects, and `UseLayoutEffect` / `UseLayoutEffectCleanup` for synchronous post-reconciliation effects. Layout effects fire synchronously after `reconcile()` within `OnComponentDirty`; regular effects are deferred via macrotask and flushed at the start of the next frame's Task Draining (React's `flushPassiveEffects` guarantee). Requires a `Destroy()` lifecycle on `ComponentNode` for unmount cleanup. | ADR 026 / TSK-062 |
| 2026-05-26 | `kitex` `UseReducer` and `UseCallback` Hooks | Added `UseReducer[S, A](reducer func(S, A) S, initial S) (func() S, func(A))` as a thin wrapper over `UseState` for complex state machines, and `UseCallback[T](callback T, deps []any) T` as a function-specialized `UseMemo` for stabilizing callback identity across re-renders. Both are purely additive to `hooks.go` with zero architectural impact. | TSK-063 |
| 2026-05-26 | `kitex` Context System (`CreateContext`, `Provider`, `UseContext`) | Introduced a React-style Context system using a stack-based provider lookup + subscription model. `CreateContext[T](defaultValue)` produces a `Context[T]` identity. `Provider` pushes a `contextEntry` onto the context's internal stack during render and pops after. `UseContext` reads from the stack on first mount (O(1)), then from a stored provider reference on re-renders (O(1)). Providers maintain subscriber sets and mark consumers dirty on value change, bypassing automatic memoization via a `!oldComp.IsDirty()` guard in `Update()`. Chosen over VDOM parent-pointer approach for superior O(1) lookup and zero per-node memory overhead. | ADR 026 / TSK-064 |
| 2026-05-26 | `kitex` Effect Engine Integration (Approach C) | Adopted React-faithful zero-engine-change integration for effect scheduling. Layout effects are self-managed by kitex, drained synchronously at the end of `OnComponentDirty` after `reconcile()`. Regular effects are queued during reconciliation; a macrotask (`engine.PostMacro`) is enqueued to call `flushPendingEffects()`. This macrotask executes at the start of the next frame's Task Draining phase (via `drainMacroTasks`). `flushPendingEffects` is also called at the top of `OnComponentDirty` before reconciliation, guaranteeing effects from frame N complete before frame N+1's reconciliation. | ADR 026 / TSK-062 |
